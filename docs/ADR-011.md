# ADR-011: Phase 9.4 — Live Templates (Structural Acceleration)

**Status:** PROPOSED  
**Date:** 2026-05-11  
**Supersedes:** None  
**Related:** ADR-009 (Phase 9 — Advanced Authoring Station), ADR-010 Extended (Identity Regeneration)

## Context

Phase 5 already introduced the foundational template grammar in the UCA model: `ModuleTemplate`, `TemplateSlotDefinition`, `OverridePolicy`, `templateRef`, `slotMappings`, and `snapshot` as the portability mechanism for frozen template instances.[cite:1] The current editor also includes an early `useTemplateCRUD` flow that injects a template instance into the tree, initializes slot mappings, and registers the template in `ui.moduleTemplates`, but it remains a lightweight insertion utility rather than a full industrial blueprint system.[cite:1][cite:2]

The existing semantics engine can already expand a node from `snapshot`, `templateRef`, or `cellRef`, apply genetic overrides, resolve slot mappings recursively, and enforce override governance through `locked`, `editable`, `hidden`, and `extendable` paths.[cite:1][cite:2] This means the conceptual substrate for parametric composition already exists in the model layer; what is missing is a production-grade authoring workflow that treats templates as first-class compositional blocks instead of static fragments.[cite:1][cite:2]

Phase 9.4 therefore formalizes **Live Templates** as a system for injecting complete, valid, reusable rack architectures into the active manifest with guided parameterization, structural validation, identity regeneration, and controlled post-merge adaptation.[cite:1][cite:2] The strategic goal is to shift the editor from node-by-node manipulation toward higher-order authoring, where the user composes functional synth structures from industrial blueprints that remain governable and portable.[cite:1][cite:2]

## Decision

Phase 9.4 will implement a **Live Templates pipeline** composed of four coordinated layers: a formal Blueprint Contract, a `BlueprintInjector` service, a Blueprint Library user interface, and an initial curated library of OMEGA Essentials modules.[cite:1][cite:2]

### 1. Blueprint Contract

Live Templates will not be stored as ad-hoc JSON fragments. They will extend the existing `ModuleTemplate` contract and keep compatibility with the current UCA template semantics, including `root`, `slots`, `policy`, `assets`, and `meta`.[cite:1][cite:2]

The contract will be expanded with the following practical authoring requirements:

- **Dynamic placeholders** for identity, naming, style, and parameter seeding, such as `${ID_PREFIX}`, `${BASE_FREQ}`, `${VOICE_NAME}`, or `${COLOR_TOKEN}`. These placeholders will be resolved before merge so the inserted structure is already concrete and auditable at commit time.
- **Compatibility metadata** that declares which target containers or rack contexts can accept the blueprint. This should align with current layout concepts such as `LayoutContainer`, presentation container mapping, tab planes, and UCA structural roles.[cite:1][cite:2]
- **Insertion schema** defining expected anchor points, allowed target kinds, default placement behavior, and whether the blueprint is root-level, container-level, or slot-bound.
- **Blueprint capabilities** flags, such as `supportsAutoWiring`, `supportsUserPrompt`, `requiresContractPorts`, and `canMaterializeSnapshot`, so the UI can accurately represent what the template can do before injection.
- **Governance inheritance** based on existing `OverridePolicy`, ensuring that locked or hidden paths remain protected even after placeholder resolution and merge.[cite:1][cite:2]

### 2. BlueprintInjector

A dedicated `BlueprintInjector` service will become the canonical structural merge engine for Live Templates. Its role is broader than the current `useTemplateCRUD`, which only mounts a template instance and leaves most industrial concerns unresolved.[cite:1][cite:2]

The injector will execute this pipeline:

1. **Resolve target context** — identify the active document, selected container, active tab, and insertion anchor.
2. **Load blueprint definition** — retrieve the template from the registry and materialize its root graph.
3. **Resolve placeholders** — substitute user-provided values and safe defaults before any merge begins.
4. **Regenerate identity recursively** — use the established ID management strategy to regenerate every internal structural ID and prevent collisions in nodes, bindings, local references, and auxiliary entities.[cite:1][cite:2]
5. **Validate structural legality pre-merge** — verify container compatibility, schema legality, parent-child constraints, grid and bounding rules, and required slot mappings before touching the active manifest.[cite:1][cite:2]
6. **Apply insertion strategy** — splice the blueprint into the chosen parent, preserving layout semantics and local ordering.
7. **Run post-merge normalization** — update projections, re-resolve derived layout if needed, and optionally materialize a portable `snapshot` for export parity.[cite:1][cite:2]
8. **Commit as a single history transaction** — the entire injection must appear as one undoable operation, preserving Phase 8 history guarantees.[cite:1]

### 3. UI: Blueprint Library & HUD

The editor will expose Live Templates through a dedicated authoring surface rather than through raw JSON editing. The UI package will include:

- **Blueprint Library side panel** with categories such as Oscillators, Filters, Modulation, FX Chains, Percussion, Utility, and Composite Voices.
- **Live preview / ghosting** so the user can inspect the rough shape and footprint of a template before committing it into the rack.
- **Parameterized injection dialog** that appears on click or drop when placeholders require user input. This dialog should resolve variables before merge and present validation hints inline.
- **Drag and drop insertion** into compatible containers, with visible acceptance or rejection states according to compatibility metadata and structural rules.
- **HUD feedback** for insertability, pending validation issues, auto-wiring decisions, and final injection status.

### 4. Initial Library: OMEGA Essentials

Phase 9.4 will ship with a curated set of high-value industrial templates so the system is immediately useful on day one. The first wave will include at minimum:

- **Classic Voice** — oscillator to low-pass filter to VCA with ADSR envelope.[cite:1]
- **Parallel FX Rack** — reverb and delay in parallel with shared wet/dry control.
- **Modulation Hub** — one or more LFO or modulation sources mapped to common destinations.
- **Utility Macro Block** — grouped controls, attenuators, and routing helpers.

These initial modules should be authored as canonical `ModuleTemplate` assets so they remain portable through the same `snapshot` and `congealSnapshot` pathways already used by bundle export and template freezing.[cite:1][cite:2]

## Auto-Wiring Decision

**Auto-Wiring will remain in scope, but only as a constrained Tier 1 capability for Phase 9.4.**[cite:1][cite:2]

The current architecture already supports explicit slot mappings and semantic binding replacement, which makes deterministic injection feasible when the target bindings are known.[cite:1][cite:2] However, fully automatic cable intelligence across arbitrary user racks would introduce ambiguity, hidden side effects, and brittle heuristics too early in the lifecycle.[cite:1][cite:2]

Therefore, Auto-Wiring is approved only in this restricted form for the first cut:

- Allow auto-binding to **well-known global targets** such as `Master_Out`, `Global_Clock`, or equivalent canonical infrastructure nodes when they are uniquely identifiable.
- Allow auto-binding when the blueprint declares a **single explicit semantic target** and the host document provides exactly one compatible match.
- Require user confirmation whenever multiple plausible targets exist.
- Record all auto-wiring decisions in the operation report so the user can audit what was connected.
- Reject silent heuristic wiring across ambiguous topologies.

Anything beyond this should be deferred to a later subphase once semantic graph confidence, telemetry, and author trust are proven in production.

## User Templates Decision

**User-defined Blueprints should be included, but as Phase 9.4B rather than day-one core scope.**[cite:1][cite:2]

The model already supports portable snapshots and recursive congealing for template-backed nodes during export, which provides a strong technical basis for “save selection as blueprint” in a later increment.[cite:1][cite:2] However, introducing user-authored templates in the first cut would combine authoring UX, serialization rules, placeholder extraction, identity rewriting, governance authoring, and local persistence in a single delivery, raising risk unnecessarily.[cite:1][cite:2]

The recommended scope split is:

- **Phase 9.4A** — system templates only, curated and validated by the product.
- **Phase 9.4B** — user can capture part of a rack as a local blueprint, edit metadata, define placeholder prompts, and save to a local library.

This preserves forward compatibility without overloading the initial milestone.

## Consequences

### Positive

- **Composition speed** increases substantially because users can inject functioning macro-architectures with one action instead of assembling each node manually.[cite:1][cite:2]
- **Consistency** improves because curated templates reuse proven topology, style, and governance patterns already aligned with the UCA system.[cite:1][cite:2]
- **Portability** remains strong because template instances can still be frozen into snapshots for export bundles and long-term reproducibility.[cite:1][cite:2]
- **Undo/redo safety** is preserved if each injection is committed as one history transaction instead of many scattered edits.[cite:1]

### Negative

- **Merge complexity** increases because insertion now touches identity, structural validation, layout reconciliation, optional auto-wiring, and history integration in one operation.[cite:1][cite:2]
- **User trust risk** appears if auto-wiring behaves opaquely or if a template mutates the host rack in ways the user did not anticipate.[cite:1][cite:2]
- **Authoring overhead** grows because curated templates need metadata, validation, governance rules, previews, and maintenance rather than just raw structure.[cite:1][cite:2]
- **Performance pressure** may rise when large blueprints trigger deep validation, layout recalculation, and projection updates on insertion.[cite:1][cite:2]

## Implementation Roadmap

## Phase 9.4A — Foundational Live Templates

### 1. Contract hardening
- [ ] Extend `ModuleTemplate` metadata to support compatibility, insertion strategy, placeholder definitions, and capability flags.[cite:1][cite:2]
- [ ] Define a serializable `BlueprintPlaceholderDefinition` contract with type, default, label, validation rules, and optional UI hints.
- [ ] Add compatibility descriptors for allowed target kinds, allowed container variants, and tab/rack context.
- [ ] Preserve backward compatibility with existing template resolution logic so legacy templates continue to resolve.[cite:1][cite:2]

### 2. Injection engine
- [ ] Implement `BlueprintInjector` as the single canonical merge service.
- [ ] Integrate recursive ID regeneration through the identity governance layer referenced by ADR-010 Extended.
- [ ] Add pre-merge structural validation against UCA legality, container compatibility, and required slot completeness.[cite:1][cite:2]
- [ ] Commit injections as a single history transaction compatible with Phase 8 multi-document undo/redo.[cite:1]
- [ ] Materialize `snapshot` optionally for portability-sensitive flows and exports.[cite:1][cite:2]

### 3. UI enablement
- [ ] Build `BlueprintLibraryPanel` with categorized browsing and search.
- [ ] Build `BlueprintPreviewCard` and rack ghost overlay.
- [ ] Build `BlueprintPromptDialog` for placeholder entry and validation.
- [ ] Add drag-and-drop insertion affordances and target highlighting.
- [ ] Add injection telemetry HUD: ready, invalid target, resolving, merged, partial auto-wire, failed.

### 4. Initial curated content
- [ ] Deliver OMEGA Essentials v1 library.
- [ ] Validate every shipped blueprint against the schema and a reference insertion matrix.
- [ ] Add industrial tags, preview metadata, and short descriptions for all bundled templates.

## Phase 9.4A.1 — Safe Auto-Wiring

- [ ] Introduce a semantic registry of canonical infrastructure nodes such as clock, MIDI in, and master output.
- [ ] Implement deterministic one-to-one auto-binding rules only for uniquely matched targets.
- [ ] Emit a merge report that lists every binding created automatically.
- [ ] Add explicit fallback to manual mapping when ambiguity exists.

## Phase 9.4B — User Blueprints

- [ ] Support “Save selection as Blueprint” from the active rack.
- [ ] Generate a congealed portable structure from the selected subtree using existing snapshot semantics.[cite:1][cite:2]
- [ ] Allow user editing of metadata, placeholder prompts, compatibility rules, and preview information.
- [ ] Persist user templates locally or in workspace storage once the editor persistence strategy is finalized.
- [ ] Add import/export for blueprint packs.

## Validation Criteria

1. **Structural legality:** Injecting a compatible blueprint never produces an invalid manifest or orphaned layout references.[cite:1][cite:2]
2. **Identity safety:** A blueprint can be injected repeatedly into the same document with zero internal ID collisions.[cite:1][cite:2]
3. **History integrity:** One injection equals one undo step, with no corruption of Phase 8 history behavior.[cite:1]
4. **Governance fidelity:** Locked and hidden paths remain enforced after placeholder resolution and merge.[cite:1][cite:2]
5. **Preview correctness:** Ghost preview footprint matches the final inserted structure within accepted layout tolerance.
6. **Auto-wiring safety:** No ambiguous auto-connection is applied without explicit confirmation.
7. **Portability:** Exported bundles preserve injected structures correctly through snapshot congealing where required.[cite:1][cite:2]

## Recommended Scope Adjustments

The proposed direction is solid and should be preserved almost as-is.[cite:1][cite:2] The main adjustment is to split ambition into two delivery bands: keep **Auto-Wiring** narrowly deterministic for the first cut, and move **user-defined Blueprints** into a follow-up increment that leverages the already existing snapshot portability model without blocking the initial release.[cite:1][cite:2]

A further refinement is to treat compatibility metadata as mandatory rather than optional. The editor already distinguishes structural containers, tab planes, and UCA hierarchy semantics, so template insertion should never rely on informal assumptions about where a blueprint can land.[cite:1][cite:2]

## Design Notes for ADR-011 Approval

This ADR should be approved with three explicit guardrails:

- **No blind structural merge** — every insertion passes through compatibility and schema validation first.[cite:1][cite:2]
- **No opaque auto-wiring** — the first release only performs deterministic, auditable connections.[cite:1][cite:2]
- **No fragmented history writes** — the injection operation is atomic from the user’s perspective and fully reversible.[cite:1]

With those constraints, Live Templates become a natural continuation of the UCA template system already present in the codebase rather than a parallel feature branch.[cite:1][cite:2]
