# ADR-009: Phase 9 — Advanced Authoring Station

**Status:** ACTIVE  
**Date:** 2026-05-11  
**Supersedes:** ADR-008 (Phase 8 History Engine)

## Context

Phase 8 established the foundations for multi-document authoring through `useDocumentOrchestrator`, including per-document undo/redo, dirty tracking based on manifest hashing, and session persistence.[cite:333] The current editor stack also centralizes authoring logic in `useManifestEditor`, which composes document state, auditing, entity CRUD, file operations, and deployment workflows behind a single hook surface.[cite:333]

The runtime side remains intentionally transitional. The current `wasmRuntime` behaves mostly as a mock bridge: it supports immediate numeric updates through `setParameter(id, value)`, provides telemetry access through `getTelemetry(id)`, and exposes `deployManifest(manifest)` for full structural synchronization, but it does not yet provide an `updateManifest` method for partial structural patching.[cite:333] That constraint makes a naive "push every edit into the runtime" strategy both inefficient and operationally unsafe.[cite:333]

Additional Phase 8 hardening analysis identified five relevant risks before moving into advanced authoring: unversioned localStorage session keys, a race condition between debounced dirty hashing and deployment hashing, deep snapshot cost under large manifests, incomplete deep ID regeneration during clipboard-style insertion, and a blocking integrity check pattern that will interfere with automatic hot-reload flows.[cite:334][cite:335]

Phase 9 therefore formalizes the Manifest Editor as a **Pro-Tier Authoring Station** by introducing a synchronization bridge between authoring state and simulation runtime, structural reconciliation tooling, and accelerated blueprint-based composition, while preserving the guarantees already established by Phase 8.[cite:333][cite:335]

## Decision Drivers

The following forces drive this ADR:

- Authoring feedback must become meaningfully faster without saturating the runtime bridge or destabilizing audio simulation behavior.[cite:333][cite:335]
- Runtime synchronization must distinguish between parameter edits and structural edits because the runtime API already supports the former granularly but only supports the latter through full manifest deployment.[cite:333]
- Undo/redo, dirty tracking, and deploy integrity must remain coherent even while synchronization becomes more automatic.[cite:333][cite:334]
- Phase 9 features should build on existing architecture rather than bypass it; `useDocumentOrchestrator` remains the source of truth for document state, while new bridge logic should be introduced as an explicit layer instead of leaking runtime behavior into low-level CRUD hooks.[cite:333][cite:335]
- Future tooling such as structural diff and blueprint injection requires stronger identity management, snapshot discipline, and separation of concerns than the current monolithic `useManifestEditor` interface provides.[cite:333][cite:335]

## Decision

Phase 9 will be implemented as a three-stream evolution of the authoring station:

1. **Phase 9.1 — WASM Hot-Reload (Simulation Sync):** Introduce a `useSimulationBridge` hook that observes the active manifest and routes changes either to immediate parameter updates (`setParameter`) or to debounced structural deployments (`deployManifest`).[cite:333]
2. **Phase 9.2 — Visual Diff & Merge:** Add a structural, entity-aware diff engine and a split-pane viewer integrated with Phase 8 history snapshots, replacing simplistic whole-document JSON comparison approaches.[cite:333][cite:335]
3. **Phase 9.3 — Advanced Templating (Blueprint Injection):** Formalize blueprint/template injection with validated schemas, collision-safe ID regeneration, and a reusable library UI for rapid composition of common synth structures.[cite:333][cite:335]

The architectural centerpiece of this ADR is `useSimulationBridge`, which becomes the runtime synchronization layer between React authoring state and the WASM worker boundary.[cite:333] The bridge will not mutate source-of-truth editor state. Instead, it will subscribe to editor changes, classify them, sequence runtime work, and surface synchronization status back to the workbench header and diagnostics log.[cite:333]

## Architecture

### 9.1 Runtime Sync Model

`useSimulationBridge` will be introduced as a dedicated bridge hook owned by the editor shell layer, not by low-level CRUD hooks.[cite:333][cite:335] Its responsibilities are:

- Observe the active document manifest, contract, and runtime readiness state.[cite:333]
- Classify edits as either **parameter-level** or **structural**.[cite:333]
- Dispatch parameter changes immediately with `wasmRuntime.setParameter()` when a user modifies a bound numeric control and the edit is runtime-safe.[cite:333]
- Queue structural changes for debounced `wasmRuntime.deployManifest()` execution when the UCA tree, containers, names, bindings, topology, or template structure changes.[cite:333]
- Expose synchronization state to the UI as `in-sync`, `syncing`, `degraded`, or `error`.[cite:333]
- Coordinate with integrity and history workflows so that runtime updates do not race with undo/redo or snapshot capture.[cite:333][cite:334]

The bridge exists because the runtime currently lacks partial structural patching through `updateManifest`; as a result, all topology-level changes must still converge through full deployment.[cite:333] Parameter updates, however, already map cleanly to the existing runtime contract because `useRackSimulation` calls `wasmRuntime.setParameter(id, value)` directly for numeric UI interactions.[cite:333]

### Change Classification

Edits will be split into two categories:

| Change class | Examples | Runtime path | Latency target |
|---|---|---|---|
| Parameter | Slider move, knob drag, numeric value update on a bound control | `setParameter(id, value)` immediately [cite:333] | Near-zero / frame-level [cite:333] |
| Structural | Rename entity, move node in UCA tree, add/remove entity, modulation rewiring, container edits, template injection | Debounced `deployManifest(manifest)` [cite:333] | Under 500ms after edit burst [cite:335] |

This split is the key design choice for Phase 9.1 because it respects the current runtime API instead of pretending patch-level structural sync already exists.[cite:333]

### Debouncing and Queueing

Structural synchronization will use a **500ms debounce window** so rapid bursts of authoring changes do not trigger repeated engine redeployments.[cite:335] The bridge will maintain an internal deployment queue and a concurrency lock to guarantee that only one structural deployment is active at a time.[cite:335]

The queue strategy is:

- Mark bridge status as `syncing` as soon as a structural change is detected.[cite:335]
- Reset the 500ms timer if new structural changes continue to arrive.[cite:335]
- When the debounce settles, snapshot the latest manifest reference and perform one `deployManifest()` call with the newest state.[cite:333]
- If changes arrive while a deployment is in flight, mark the queue as dirty and schedule exactly one follow-up deployment with the latest manifest once the current one completes.[cite:335]
- Do not attempt mid-flight cancellation because the runtime API does not currently expose abort semantics.[cite:333]

### Integrity Locks and Race Avoidance

Phase 8 analysis identified a high-priority race between debounced dirty hashing and deployment hashing, especially in flows such as undo → modify → deploy inside a short time window.[cite:334] To address this, Phase 9.1 adopts the following constraints:

- `useSimulationBridge` must flush or await pending dirty-hash debounce work before any structural deployment decision is finalized.[cite:334]
- `useDocumentOrchestrator` should track the currently active debounce timer and expose a safe synchronization point for deployment-oriented flows.[cite:333][cite:334]
- The bridge must honor **history locks** during undo/redo so runtime synchronization does not deploy an intermediate state while the document reducer is still reconciling snapshots.[cite:335]
- Automatic hot-reload deployments must not use the same blocking integrity confirmation flow as manual deployment, because current confirmation-driven coherence checks would produce false-positive interruptions during rapid editing.[cite:334]

For this reason, integrity validation will be split into two modes:

| Mode | Trigger | Behavior |
|---|---|---|
| Manual deploy | Explicit user deployment action | Full integrity check, including mismatch confirmation when contract fingerprint diverges.[cite:333] |
| Auto hot-reload | Bridge-driven structural sync | Non-blocking validation; log warning and surface degraded state instead of interrupting the editing loop.[cite:334][cite:335] |

### Sync Status UI

A runtime synchronization indicator will be added to `MultiTabHeader` so the author always sees the current simulation state.[cite:335] The indicator states are:

- 🟢 **In Sync** — runtime reflects the latest known editor state.[cite:335]
- 🟡 **Syncing...** — parameter or structural updates are pending or in flight.[cite:335]
- 🔴 **Error** — last runtime deployment failed, integrity is degraded, or runtime is unavailable.[cite:335]

The HUD may also expose compact metadata such as last successful deploy timestamp, pending structural update count, and current mode (`live`, `degraded`, `manual-only`).[cite:335]

### Error Recovery

Because `wasmRuntime.deployManifest()` currently simulates asynchronous deployment with non-trivial latency, the bridge must be resilient to failures and re-entry.[cite:333] The recovery contract is:

- Deployment errors move the status to `error` and append structured diagnostics to the log stream.[cite:333][cite:335]
- The runtime bridge retains the latest manifest intent in memory so the next successful retry converges to the newest state rather than replaying stale intermediate payloads.[cite:335]
- A bounded retry policy with backoff may be used for transient runtime failures, but manual user actions remain able to force resynchronization.[cite:335]

## Detailed Roadmap

### Phase 9.1 — The Live Loop

#### Objective

Create a bidirectional-feeling authoring loop in which editor actions are reflected in simulation fast enough to feel live, without changing the current document authority model or requiring a new partial-patch runtime API.[cite:333][cite:335]

#### Scope

- New `useSimulationBridge` hook.[cite:335]
- Runtime-safe change classifier for parameter vs structural edits.[cite:333]
- Debounced structural deploy queue (500ms).[cite:335]
- Lock-aware synchronization around undo/redo and dirty tracking.[cite:334][cite:335]
- `MultiTabHeader` sync status HUD.[cite:335]
- WASM crash recovery and degraded-mode behavior.[cite:335]

#### Implementation Plan

##### Workstream 1: Hardening prerequisites

Before enabling automatic runtime sync, the following must be completed:

- Version localStorage session keys to prevent silent restore corruption during schema evolution.[cite:334]
- Flush or coordinate dirty-hash debounce work before runtime deployment decisions.[cite:334]
- Introduce a stable `hashInProgress` or equivalent guard to prevent overlapping hash calculations from producing stale `isDirty` state.[cite:334]
- Preserve deep identity regeneration for future blueprint injection so runtime sync does not amplify broken structural references introduced by copy/paste flows.[cite:334][cite:335]

##### Workstream 2: Bridge foundation

Implement `useSimulationBridge` with an interface similar to:

```ts
interface SimulationBridgeState {
  status: 'idle' | 'syncing' | 'in-sync' | 'degraded' | 'error';
  pendingStructuralSync: boolean;
  lastSuccessfulSyncAt: number | null;
  lastError: string | null;
  pushParameterUpdate: (id: string, value: number) => void;
  scheduleStructuralSync: (reason: string) => void;
  forceResync: () => Promise<void>;
}
```

This hook will consume the active manifest, contract, runtime readiness, and optional editor lock state from the shell level around `useManifestEditor`.[cite:333][cite:335]

##### Workstream 3: Parameter fast path

Wire numeric control updates to the bridge so runtime-safe parameter edits bypass full deployment and flow directly through `setParameter`.[cite:333] This path should be used only when all of the following hold:

- The edited entity resolves to a runtime parameter binding.[cite:333]
- The change is numeric and does not alter structure, topology, or identity.[cite:333]
- The runtime is loaded and not in a terminal error state.[cite:333]

This path should preserve the current low-latency behavior already present in `useRackSimulation`, but route it through a centralized synchronization layer so the runtime state model and editor sync HUD stay coherent.[cite:333]

##### Workstream 4: Structural sync queue

All structural mutations route to `scheduleStructuralSync(reason)` rather than calling `deployManifest()` inline.[cite:333][cite:335] Candidate structural sources include:

- Entity add/remove/duplicate/paste.[cite:333]
- Layout container add/update/remove.[cite:333]
- Modulation graph changes.[cite:333]
- UCA tree mutations and template injection.[cite:333]
- Name, bind, role, component, and presentation changes that affect runtime interpretation.[cite:333]

##### Workstream 5: UI and observability

Add a compact simulation status element in `MultiTabHeader` and a richer diagnostics entry in logs or a future diagnostics panel.[cite:335] The indicator should display current sync state without becoming noisy, and should never block the user from continuing to edit unless the runtime enters a fatal failure mode.[cite:335]

#### Acceptance Criteria

- Parameter edits propagate to the runtime with no perceptible debounce overhead for supported controls.[cite:333]
- Structural authoring bursts settle into one deployment after 500ms of inactivity.[cite:335]
- No UI stalls occur during automatic runtime sync under normal authoring load.[cite:335]
- Undo/redo does not produce out-of-order runtime states.[cite:334][cite:335]
- Manual deploy remains available and uses strict integrity confirmation, while auto-sync uses non-blocking degraded-state reporting.[cite:333][cite:334]

### Phase 9.2 — Visual Reconciliation

#### Objective

Provide history-aware structural comparison so authors can inspect what changed between snapshots, current state, or prior revisions without relying on opaque whole-document string comparisons.[cite:333][cite:335]

#### Scope

- Tree-aware diff algorithm with entity-level and property-level comparison.[cite:335]
- `DiffViewer` split-pane UI component.[cite:335]
- History integration for “compare with current” and “compare with previous” modes.[cite:335]
- Highlighting for additions, removals, and modifications across the UCA tree and projected legacy sections.[cite:335]

#### Implementation Notes

The current history mechanism stores deep-cloned snapshots using `JSON.parse(JSON.stringify(manifest))`, which is serviceable for correctness but costly for large manifests.[cite:333][cite:334] Before building heavy visual reconciliation, the implementation should either introduce structural sharing or compute diffs lazily so large histories do not freeze the UI during comparison workflows.[cite:334][cite:335]

#### Acceptance Criteria

- The diff engine correctly distinguishes added, removed, and modified nodes on a representative 200-node manifest.[cite:335]
- Comparison mode opens directly from history entries without mutating active document state.[cite:335]
- Large-manifest diffing remains responsive enough for production use and does not regress Phase 8 history behavior.[cite:334][cite:335]

### Phase 9.3 — Structural Acceleration

#### Objective

Enable rapid prototyping by injecting reusable blueprint structures into the active UCA tree with predictable identity handling and validation.[cite:333][cite:335]

#### Scope

- Blueprint schema definition with placeholders.[cite:335]
- Template library storage and browsing UI.[cite:335]
- Injection logic with deep ID regeneration and collision protection.[cite:334][cite:335]
- Built-in standard module chains for common authoring scenarios.[cite:335]

#### Implementation Notes

The current `useTemplateCRUD` already demonstrates the direction of blueprint-style injection by appending a container node based on a `ModuleTemplate` and registering it into `moduleTemplates`.[cite:333] However, current clipboard and paste flows only regenerate top-level IDs and do not comprehensively rewrite nested attachments or cross-node references, which is inadequate for a full blueprint system.[cite:333][cite:334] Deep ID regeneration therefore becomes a formal prerequisite for Phase 9.3.[cite:334][cite:335]

#### Acceptance Criteria

- Blueprint injection never collides with existing IDs in the active document.[cite:334][cite:335]
- Nested references remain valid after injection.[cite:334]
- Injected structures are fully undoable within the Phase 8 history model.[cite:333][cite:335]

## Consequences

### Positive

- **Faster design loop:** Parameter updates become effectively immediate while structural sync remains stable and bounded.[cite:333][cite:335]
- **Cleaner architecture:** Runtime synchronization moves out of `useManifestEditor` into an explicit bridge layer, reducing orchestration sprawl.[cite:333][cite:335]
- **Safer extensibility:** Diff and blueprint features build on stronger identity, queueing, and synchronization primitives rather than ad hoc editor mutations.[cite:334][cite:335]
- **Better operator trust:** Persistent sync status makes runtime/editor divergence visible instead of implicit.[cite:335]

### Negative

- **Higher orchestration complexity:** Automatic synchronization introduces locks, queue semantics, degraded modes, and additional editor/runtime state coordination.[cite:334][cite:335]
- **Runtime API pressure:** The absence of `updateManifest` keeps structural sync heavier than ideal and may motivate a future incremental runtime API.[cite:333]
- **Performance risk on large documents:** Visual diff and history-heavy workflows still require snapshot cost mitigation to scale cleanly past large manifests.[cite:334][cite:335]

## Rejected Alternatives

### Push every edit through `deployManifest()`

Rejected because the runtime currently only supports full structural deployment, which is too coarse for high-frequency authoring interactions and would lead to unnecessary churn during typing, dragging, or repeated layout edits.[cite:333][cite:335]

### Add a partial structural patch layer immediately

Rejected for Phase 9.1 because the runtime API does not yet expose `updateManifest`, and inventing an editor-only patch protocol now would create complexity without a real runtime counterpart.[cite:333]

### Keep direct `setParameter()` calls scattered across UI hooks

Rejected because Phase 9 requires centralized observability, error handling, and status reporting. The current direct-call approach works for isolated simulation value updates but does not provide a coherent runtime synchronization model for the broader authoring station.[cite:333][cite:335]

## Implementation Order

| Order | Work item | Reason |
|---|---|---|
| 1 | Hardening fixes from Phase 8 analysis | Removes blockers before automation amplifies them.[cite:334][cite:335] |
| 2 | `useSimulationBridge` foundation | Establishes the new runtime boundary and sync contract.[cite:335] |
| 3 | Sync HUD in `MultiTabHeader` | Makes the bridge observable and debuggable in daily use.[cite:335] |
| 4 | Manual vs auto integrity split | Prevents hot-reload from inheriting blocking deployment behavior.[cite:334] |
| 5 | Visual diff engine | Builds on stable history and runtime separation.[cite:334][cite:335] |
| 6 | Blueprint library and injection | Builds on identity hardening and structural tooling.[cite:334][cite:335] |

## Validation Criteria

1. **WASM sync:** Numeric parameter edits propagate immediately where runtime-safe, and structural edits converge in under 500ms after an edit burst without visible UI stutter.[cite:333][cite:335]
2. **History safety:** Automatic synchronization never breaks Phase 8 undo/redo guarantees or captures incorrect dirty state during rapid edit/deploy sequences.[cite:333][cite:334]
3. **Diff accuracy:** Structural comparison correctly identifies additions, removals, and property mutations on representative medium-size manifests.[cite:335]
4. **Template integrity:** Blueprint injection preserves internal relationships, rewrites IDs safely, and remains fully reversible through history.[cite:334][cite:335]
5. **Operational clarity:** The header-level sync HUD always reflects whether the runtime is synchronized, pending, degraded, or failed.[cite:335]

- Create the implementation epic for **Phase 9.1 — The Live Loop** with sub-tasks for hardening, bridge foundation, queueing, HUD, and error recovery.[cite:335]
- Refactor `useManifestEditor` incrementally so deployment behavior moves behind the new bridge API instead of remaining embedded in the orchestration hook.[cite:333][cite:335]
- Add regression coverage for undo/redo, dirty tracking, clipboard/template insertion, and runtime synchronization sequencing.[cite:333][cite:334]
- Revisit runtime API design after Phase 9.1 to evaluate whether a future `updateManifest` method is justified by real hot-reload usage patterns.[cite:333]
