# ADR-010: Phase 9.3 Selective Merge Engine

## Context and Problem Statement
Following the successful implementation of the **Visual Diff Engine (Phase 9.2)**, the OMEGA Manifest Editor requires a mechanism to selectively apply changes from historical snapshots back into the active document. Users need the ability to "pick" specific additions, removals, or property modifications without performing a destructive full rollback.

## Decision Drivers
*   **Identity Integrity**: Merge operations must preserve the canonical ID system to prevent node duplication or broken bindings.
*   **Undo/Redo Stability**: Every merge operation must be recorded as a new history entry to allow safe reversal.
*   **Runtime Synchronization**: Merged changes must trigger a `StructuralSync` via the `simulationBridge` to update the active rack/audio state.
*   **Granularity**: Support for individual `DiffEntry` reconciliation (property-level vs entity-level).

## Proposed Strategy: Identity-First Selective Merge

### 1. Merge Granularity
We will implement **Entry-Level Reconciliation**. Each `DiffEntry` generated by the engine can be applied independently.
*   **Added Entity**: Injects the entity into the target manifest and regenerates child IDs if it's a UCA node.
*   **Removed Entity**: Deletes the entity based on its ID.
*   **Modified Property**: Applies the specific `fieldPath` value from the source to the target.

### 2. Safety Invariants
*   **Reference Guard**: An entity cannot be added if its parent container ID is missing in the target manifest (dependency check).
*   **Uniqueness Guard**: Prevent adding an entity if an ID collision exists (require regeneration or replace).
*   **Atomic History**: A "Bulk Merge" operation (applying multiple entries) must result in a **single** history snapshot labelled `[MERGE] Selective reconciliation from history`.

### 3. Implementation Plan
1.  **Engine Extension**: Add `applyMerge(target, diffEntry)` utility in `manifestDiff.ts`.
2.  **Hook Integration**: Implement `handleMergeEntries(entries[])` in `useManifestEditor.ts`.
3.  **UI Interaction**: Update `ManifestDiffViewer.tsx` with "Apply" buttons for each change row.
4.  **Batch Reconciliation**: Support "Apply All" with conflict resolution logic.

## Considered Alternatives

### 1. Entity-Level Full Replace
*   **Pros**: Simpler implementation; avoids field-level patching.
*   **Cons**: Destructive for concurrent property changes; violates the principle of least privilege in merging.
*   **Status**: **REJECTED** - Lacks the granularity required for industrial auditing.

### 2. Blind Batch Rollback
*   **Pros**: Extremely fast; no diff computation needed for merge.
*   **Cons**: Same as standard Undo/Redo; doesn't allow "cherry-picking" specific improvements from a discarded design path.
*   **Status**: **REJECTED** - Redundant with existing history engine.

### 3. Entry-Level Reconciliation (Chosen)
*   **Pros**: Maximum control; safe property patching; identity-first stability.
*   **Cons**: Highest implementation complexity; requires dependency checking.
*   **Status**: **ACCEPTED** - Aligns with OMEGA industrial authoring standards.

## Consequences
*   **Positive**: High-fidelity control over document evolution; no need for full rollbacks.
*   **Positive**: Standardized `DiffEntry` contract simplifies future collaborative sync (Conflict-free Replicated Data Types path).
*   **Negative**: Increased complexity in handling nested UCA dependencies during partial merges.
